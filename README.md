MES (Manufacturing Execution System)製造執行系統

專門為工廠端設計的一套生產管理系統。
負責「執行」與「監控」的雙手與神經。能將工廠生產現場的數據即時回傳，讓管理者清楚知道：現在是誰在做什麼？在哪一台機器做？用了哪些料？目前進度如何？

四大核心功能
1.	生產追溯 (Traceability)：
能詳盡記錄產品的「身分證」，包含原料批號、加工人員、使用的設備、測試數據等。當產品出現瑕疵時，能快速追溯問題來源。

2.	生產調度與派工 (Dispatching)：
根據 ERP 下達的生產計畫，MES 會將具體的工單分配到各個機台與作業員手上，並即時調整生產順序以應對突發狀況。

3.	品質管理 (Quality Management)：
即時收集生產過程中的品質檢測數據（如尺寸、重量、電壓等），一旦發現數值偏離標準，系統會立即發出警示甚至停機。

4.	設備管理 (OEE 監控)：
監控設備的稼動率、故障率與保養狀態。透過數據分析，計算出「設備綜合效率 (OEE)」，協助工廠優化生產效能。

半導體 MES 的核心差異：極致的複雜度與精確度
1.	製程回流（Re-entrant Flow）：
一般製造業的工序是線性的（A -> B -> C）。但在半導體（尤其是晶圓代工）中，晶圓會反覆回到同一台設備進行不同的塗佈、顯影、蝕刻。MES 必須精確追蹤每一片晶圓（Wafer）目前是處於第幾層、哪一個循環。

2.	自動化整合深度（SEC/GEM）：
半導體設備昂貴，為了最大化產能，MES 必須與設備透過標準通訊協定（如 SECS/GEM）高度整合。系統不只是接收數據，還能直接下令設備更換參數（Recipe），達成無人化生產。

3.	管理單位的細緻度：
一般業可能以「批」為單位，但半導體 MES 必須追蹤到每一片 Wafer 的位置，甚至是 Wafer 上的特定座標數據，以便進行良率分析。

資料庫串聯
1.	核心關聯式資料庫 (Relational Database, RDBMS)
這是 MES 的「主資料庫」，負責儲存工廠的靜態與動態營運資料。
A. 常見資料庫： Oracle, Microsoft SQL Server (MSSQL), PostgreSQL。
B. 儲存內容：
  i.	主數據： 產品規格（Spec）、配方（Recipe）、作業員權限。
  ii.	交易紀錄：工單狀態、過站紀錄（Track-in/out）、批號（Lot ID）追蹤。
C. 特點： 強調資料的 ACID（原子性、一致性、隔離性、持久性），確保生產紀錄絕對正確且不可竄改。

2. 時序資料庫 (Time-Series Database)
在半導體或自動化程度高的產業，設備每秒會產生大量的感測器數據，這類資料通常不會存入傳統 SQL，而是存入時序資料庫。
A. 常見資料庫： InfluxDB, Prometheus, AVEVA Historian (PI System)。
B. 儲存內容：
  i.	機台參數： 溫度、壓力、電壓、流量的即時波動。
  ii.	設備狀態： 稼動率、警報（Alarm）發生時間。
C. 特點： 寫入速度極快，適合存放隨時間變動的連續數據，用於後續的 SPC (統計製程管制)。

3.快取與記憶體資料庫 (In-Memory Database)
為了應對高頻率的查詢（例如驗證權限或檢查產品狀態），MES 會使用快取層。
A. 常見資料庫： Redis, Memcached。
B. 儲存內容：
  i.	當前在製品 (WIP) 的位置快取。
  ii.	使用者登入 Session。
C. 特點： 讀取延遲極低，確保工廠生產線在過站掃描時不會因為資料庫查詢太慢而停滯。

4. 巨量資料與分析平台 (Big Data / Data Warehouse)
當 MES 累積了數年的數據後，為了進行良率分析（Yield Analysis）或機器學習模型訓練，資料會被抽取到分析平台。
A. 常見資料庫： Hadoop (HDFS), Amazon Redshift, Snowflake。
B. 儲存內容：
  i.	歷史生產趨勢分析。
  ii.	跨廠區的經營績效比較。

5. 跨系統的資料交換 (System Integration)
除了資料庫，MES 還會透過 API 或中間件與其他管理系統串聯：
A. 與 ERP 串聯： 獲取訂單與物料資訊（通常透過 SQL 轉檔或 Web Service）。
B. 與 EAP/PLC 串聯： 透過 SECS/GEM 通訊協定或 OPC-UA 伺服器，將底層設備的 raw data 拋轉回資料庫。

缺點
1.	改動困難： 傳統 MES 的架構通常非常龐大且緊密。如果今天產線要增加一條新的生產邏輯（例如新的測試流程），修改程式碼的風險很高，常被稱為「牽一髮而動全身」。
2.	非標業務難適配： 許多 MES 是「標準化軟體」，但每間工廠的眉角（KNOW-HOW）都不同，常導致「軟體跟不上業務變化」的窘境。
3.	人才斷層： 運維 MES 的人需要同時懂「軟體技術」與「工廠製程」。如果這類關鍵工程師離職，系統出問題時往往沒人敢動。

→ 容器化 (Docker/K8s)： 開始應用在 MES 的微服務架構中，讓系統可以「局部升級」而不必整台關機。
